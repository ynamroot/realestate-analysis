## Integration Findings

### Gap 1: Geocoding Not Wired into CLI (Critical)

**Affects:** SUBW-01, COMM-05, full E2E flow

`geocode_all_apartments()` in `pipeline/collectors/geocode.py` is fully implemented but never called from `pipeline/cli/main.py::collect()`.

The subway collection step (`collect_all_subway_distances`) queries:
```sql
SELECT id, apt_nm, latitude, longitude FROM apartments WHERE latitude IS NOT NULL
```
Without a preceding geocode run, this returns 0 rows — subway and commute data collection is silently skipped.

**Fix:** Add `geocode` as a `data_type` option in `collect()`, or auto-run geocoding before subway step when `data_type=all`.

---

### Gap 2: `subway_distances` UNIQUE Constraint Mismatch (Critical)

**Affects:** SUBW-01, SUBW-02, SUBW-03

`pipeline/storage/schema.py` line 83:
```sql
UNIQUE(apartment_id, station_name)   -- 2 columns
```
`pipeline/collectors/subway_distances.py` lines 89, 105 — INSERT with `(apartment_id, station_name, line_name, walk_distance_m)`.

For multi-line stations (강남 = Line 2 + Bundang Line, 신도림 = Line 1 + 2, etc.), only the first line's record is stored. `INSERT OR IGNORE` silently discards subsequent lines.

**Fix:** Change schema to `UNIQUE(apartment_id, station_name, line_name)` + migration.
